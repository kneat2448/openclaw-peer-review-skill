# Telegram Peer Review MVP

Use when the user wants to install, migrate, debug, or extend the Telegram-native peer review workflow with SQLite storage and a custom dashboard.

## What this skill covers

- Copy or migrate the peer-review app into an OpenClaw workspace or GitHub repo.
- Configure env vars, team roster, and Telegram IDs.
- Start, restart, test, and debug the bot/dashboard.
- Explain or modify any part of the review flow.

## App shape

```text
src/
  bot.js          — all Telegram bot logic, review flow, session persistence
  db.js           — SQLite schema, migrations, helpers
  analysis.js     — scoring, risk, expectation fit, dashboard JSON generation
  scheduler.js    — cron job scheduling for review dates
  server.js       — Express server + bot bootstrap
scripts/
  check-env.js    — validates required env vars at startup
  smoke-test.js   — end-to-end smoke test
  test-cadence.js — review date scheduling unit tests
  test-dashboard-team.js — dashboard analysis integration test
dashboard/
  index.html      — single-page dashboard frontend
  public/data/    — generated per-project JSON files
data/
  team_members.json — team roster with Telegram IDs
ecosystem.config.cjs — pm2 config for persistent deployment
.env.example
package.json
```

Do not copy `package-lock.json` across machines — regenerated by `npm install`.
Do not copy `data/peer_review.db*` unless preserving existing history.

## Standard workflow

1. Read `references/setup.md` for install/migration steps.
2. Inspect `.env`, `data/team_members.json`, and `src/bot.js` before changing anything.
3. After any code change, run:

```bash
node --check src/bot.js
npm run test:cadence
npm run test:dashboard
npm run smoke
```

4. Restart: `npm start` or `pm2 restart peer-review-bot`.
5. Verify `GET /` responds and Telegram polling logs show started.

## Telegram commands

Only `TECH_LEAD_USER_ID` can use these:

- `create project` — sends one-message project setup template
- `projects` — lists all projects with status
- `start review [name or number]` — sends reviews for a specific project
- `analyze reviews [name or number]` — runs analysis, updates dashboard
- `dashboard [name or number]` — returns dashboard URL
- `status [name or number]` — shows per-reviewer submission status
- `/start` or `/help` — shows command list

All commands that take a project accept: project name (partial match), project number from `projects` list, or no arg (uses latest).

## Review flow

1. Tech lead sends `create project` → bot sends one-message setup template.
2. Tech lead fills all fields and replies → project + schedule saved to SQLite.
3. At scheduled time or `start review` → bot DMs each real team member asking "yes / later".
4. Reviewer replies "yes" → bot sends a single 15-question questionnaire.
5. Reviewer replies with all answers in one message (numbered `1:` through `15:`).
6. Bot validates, stores, moves to next teammate, or marks complete.
7. Tech lead sends `analyze reviews` → scores computed, dashboard JSON written.
8. Dashboard served at `BASE_URL/dashboard/<project_id>`.

## Key behaviors

- **Session persistence**: reviewer sessions stored in `reviewer_sessions` DB table — survive process restarts.
- **Send failure handling**: per-reviewer try/catch; failures logged to `review_send_log`; tech lead notified of unreachable members.
- **Multi-project**: all commands accept project name or number; `projects` lists all.
- **Partial answer preservation**: on validation failure, valid answers pre-fill the re-sent questionnaire.
- **"Later" reminders**: reviewer saying "later" triggers a reminder after `LATER_REMIND_MS` (default 4h). A 5-min loop checks for due reminders.
- **Per-member status**: `status` shows each reviewer's state (submitted / in progress / waiting / not started).
- **Score tolerance**: accepts `8`, `7.5`, `8/10`, `7.5/10`, `8 out of 10`.
- **Empty response display**: members with 0 responses show "No responses" on dashboard instead of `0/10`. Risk shows `Unknown`.
- **Sensitive data isolation**: contracts/offers/terms stored in `project_sensitive_data`, excluded from analysis and dashboard.
- **Placeholder IDs**: members without `telegram_user_id` get `member:<projectId>:N` — tech lead receives their review instead.

## DB tables

| Table | Purpose |
|---|---|
| `projects` | Project metadata |
| `project_sensitive_data` | Contracts, offers, terms (excluded from dashboard) |
| `team_members` | Per-project member roster |
| `review_sessions` | Sent review sessions |
| `reviewer_sessions` | Persistent per-reviewer in-progress state |
| `review_send_log` | Per-reviewer send success/failure log |
| `responses` | Submitted review answers |
| `results` | Analyzed scores and summaries |
| `cron_jobs` | Scheduled review jobs |

## Env vars

| Var | Required | Default | Notes |
|---|---|---|---|
| `TELEGRAM_BOT_TOKEN` | yes | — | From BotFather |
| `TECH_LEAD_USER_ID` | yes | — | Telegram user ID (numbers only) |
| `BASE_URL` | yes | — | Public URL for dashboard links |
| `PORT` | no | 3000 | Express server port |
| `DB_PATH` | no | `./data/peer_review.db` | SQLite file path |
| `APP_TIME_ZONE` | no | `Asia/Kolkata` | Timezone for scheduled review display |
| `LATER_REMIND_MS` | no | `14400000` (4h) | Milliseconds before "later" reminder fires |

## Common fixes

- **Bot not responding**: check one poller per token; check `TECH_LEAD_USER_ID` matches sender; check process is running.
- **Wrong scheduled time**: set `APP_TIME_ZONE` in `.env`.
- **Reviews not reaching members**: add `telegram_user_id` to `data/team_members.json`; each user must have started the bot once.
- **Reviewer lost their session**: process restart now safe — sessions are in DB.
- **DB size**: WAL grows with test runs. See cleanup commands in `references/setup.md`.

## BASE_URL in local environments

Use localtunnel or ngrok for a public URL:

```bash
npx localtunnel --port 3000
# or
ngrok http 3000
```

Update `BASE_URL` in `.env` and restart. URL changes each session — update it or use a persistent tunnel.
